Automated Changelog from GitHub Releases

A complete guide to automatically generating beautiful changelogs from your GitHub Releases with GitHub Actions, tools, and best practices.

📅 April 10, 2026 ⏱️ 12 min read 🛠️ Technical Guide

Keeping your changelog up-to-date is essential for SaaS products, but manually copying release notes between GitHub and your public changelog is tedious and error-prone. What if you could automatically generate a beautiful changelog from your GitHub Releases?

In this comprehensive guide, you'll learn how to set up automated changelog generation from GitHub Releases, compare the best tools for the job, and implement best practices used by top open-source projects.

What Are GitHub Releases?

GitHub Releases are a way to package and provide software to your users along with release notes and binary files. They're tightly integrated with Git tags and are the standard way to announce new versions of your software.

Key features of GitHub Releases:

  • Version tagging — Associated with Git tags (e.g., v1.2.0)
  • Release notes — Markdown descriptions of changes
  • Asset attachments — Binary files, installers, documentation
  • Pre-release markers — Distinguish stable from beta releases
  • Automatic changelogs — GitHub can auto-generate from PRs and commits
💡 Why Use GitHub Releases for Changelogs?

GitHub Releases are where developers already document changes. By automatically syncing them to a public changelog page, you eliminate duplicate work and ensure your users always see the latest updates without manual copying.

Method 2: GitHub Actions + Static Site

If you prefer a DIY approach, you can build your own automated changelog using GitHub Actions to generate a static site.

Step-by-Step Setup

1. Create the GitHub Action Workflow

YAML 
# .github/workflows/generate-changelog.yml
name: Generate Changelog

on:
  release:
    types: [published]
  workflow_dispatch:

jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: write
      pages: write
    
    steps:
      - name: Checkout
        uses: actions/checkout@v4
      
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
      
      - name: Fetch releases
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          mkdir -p changelog
          gh release list --limit 100 --json name,tagName,publishedAt,body \
            | node scripts/generate-changelog.js
      
      - name: Build site
        run: |
          # Build your static site here
          npm run build
      
      - name: Deploy to GitHub Pages
        uses: actions/deploy-pages@v4

2. Create the Changelog Generator Script

JavaScript 
// scripts/generate-changelog.js
const fs = require('fs');
const path = require('path');

// Read releases from stdin (piped from gh CLI)
let input = '';
process.stdin.on('data', chunk => input += chunk);
process.stdin.on('end', () => {
  const releases = JSON.parse(input);
  
  // Sort by date (newest first)
  releases.sort((a, b) => new Date(b.publishedAt) - new Date(a.publishedAt));
  
  // Generate HTML
  const html = generateChangelogHTML(releases);
  
  // Write to file
  fs.writeFileSync('changelog/index.html', html);
  console.log(`Generated changelog with ${releases.length} releases`);
});

function generateChangelogHTML(releases) {
  const entries = releases.map(r => `
    

      
${r.name}

      ${new Date(r.publishedAt).toLocaleDateString()}
      
${markdownToHTML(r.body)}

    

  `).join('');
  
  return `


  Changelog
  


  

    
Changelog

    ${entries}
  


`;
}

Best Practices for GitHub Release Changelogs

1. Write Great Release Notes

Your automated changelog is only as good as your GitHub Release notes. Follow these guidelines:

  • Use clear titles — "Fixed login bug on mobile" not "Bug fix"
  • Categorize changes — Group by Features, Bug Fixes, Improvements
  • Link to issues/PRs — Reference #123 for traceability
  • Include migration notes — For breaking changes, explain what users need to do
  • Keep it scannable — Use bullet points, not paragraphs
Markdown 
## 🚀 Features
- Added dark mode support #234
- New API endpoint for webhooks #245

## 🐛 Bug Fixes
- Fixed mobile navigation glitch #240
- Corrected timezone display in exports #238

## 📚 Documentation
- Updated API reference with examples #241

2. Use Semantic Versioning

Follow Semantic Versioning (SemVer) for your release tags:

  • MAJOR (X.y.z) — Breaking changes
  • MINOR (x.Y.z) — New features (backwards compatible)
  • PATCH (x.y.Z) — Bug fixes (backwards compatible)

This helps your automation categorize releases and helps users understand the impact of updating.

3. Handle Pre-releases

Use GitHub's pre-release feature for beta versions:

  • Mark alpha/beta/RC versions as pre-releases
  • Configure your automation to filter them from the public changelog (or show them in a separate section)
  • Use tags like v2.0.0-beta.1 for clarity
⚠️ Pre-release Considerations

Not all users want to see beta releases in your public changelog. Consider your audience—developers may appreciate seeing upcoming features, while end-users might find it confusing. Changelog.page lets you filter pre-releases with a simple configuration option.

4. Automate Release Creation

Take automation further by auto-creating releases from your Git workflow:

YAML 
# Auto-create release when version tag is pushed
on:
  push:
    tags:
      - 'v*'

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Create Release
        uses: softprops/action-gh-release@v1
        with:
          generate_release_notes: true
          prerelease: ${{ contains(github.ref, 'beta') || contains(github.ref, 'alpha') }}

Advanced: Multi-Repository Changelogs

For organizations with multiple repositories, you might want a unified changelog:

🏢 Enterprise Use Case

If you manage multiple services or packages, consider aggregating all releases into a single product changelog:

  • API service releases
  • Web dashboard updates
  • Mobile app releases
  • Documentation changes

To implement this, modify your GitHub Action to fetch releases from multiple repos:

JavaScript 
// Fetch releases from multiple repositories
const repos = ['org/api', 'org/web', 'org/mobile'];
const allReleases = [];

for (const repo of repos) {
  const releases = await fetch(`https://api.github.com/repos/${repo}/releases`)
    .then(r => r.json());
  
  allReleases.push(...releases.map(r => ({
    ...r,
    repo: repo.split('/')[1], // Tag with repo name
    sortDate: new Date(r.published_at)
  })));
}

// Sort by date across all repos
allReleases.sort((a, b) => b.sortDate - a.sortDate);

// Group by repo in your template or interleave them

Troubleshooting Common Issues

Releases Not Syncing

  • Verify your GITHUB_TOKEN has permission to read releases
  • Check that the repository name format is correct (owner/repo)
  • Ensure the workflow is triggered (check Actions tab)

Formatting Issues

  • GitHub Flavored Markdown may not render identically in your changelog—test thoroughly
  • Use a Markdown parser that supports GitHub's extensions (tables, task lists, etc.)
  • Sanitize HTML to prevent XSS if displaying user-generated content

Rate Limiting

  • Unauthenticated GitHub API requests are limited to 60/hour
  • Use GITHUB_TOKEN for 1,000 requests/hour
  • Cache release data between builds to minimize API calls
