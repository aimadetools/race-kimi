Frequently Asked Questions

Quick answers to common questions about Changelog.page. Can't find what you're looking for? Contact us.

📋 General

What is Changelog.page? ▼

Changelog.page is a static site generator that turns Markdown files into beautiful, customizable changelog pages. It's designed for SaaS companies, indie hackers, and open-source projects who want to communicate product updates to their users.

Write your updates in Markdown with YAML frontmatter, run the build command, and deploy to any static host. No database, no complex setup — just clean, fast changelogs.

Who is Changelog.page for? ▼

Changelog.page is perfect for:

  • SaaS companies wanting to share product updates
  • Indie hackers building in public
  • Open source projects announcing releases
  • Design systems documenting changes
  • API products communicating breaking changes

If you ship updates and want your users to know about them, Changelog.page is for you.

Is Changelog.page free? ▼

Yes! The core generator is open source and free to use forever. You can:

  • Generate unlimited changelogs
  • Use all three themes (Minimal, Cards, Timeline)
  • Host on any static hosting platform
  • Customize the templates however you want

We plan to offer a hosted Pro version with additional features like custom domains, private changelogs, and analytics. But the self-hosted version will always be free.

🚀 Getting Started

What do I need to get started? ▼

To use Changelog.page, you'll need:

  • Node.js 18 or higher
  • npm or yarn for package management
  • A code editor (VS Code recommended)
  • Basic familiarity with Markdown

You'll also need a hosting platform like Vercel, Netlify, or GitHub Pages to deploy your changelog. See our Getting Started guide for detailed instructions.

How long does it take to set up? ▼

Most users have their first changelog live in 5-10 minutes. Here's the typical timeline:

  1. Clone the repository (1 minute)
  2. Install dependencies (1-2 minutes)
  3. Write your first entry (3-5 minutes)
  4. Build and deploy (2 minutes)

If you're new to static site generators or deployment platforms, budget 15-20 minutes for your first setup.

Do I need to know how to code? ▼

Basic familiarity with Markdown and command line is helpful, but you don't need to be a developer. Writing changelog entries is just like writing in any text editor.

If you can write a README file, you can use Changelog.page. The build process is a single command: npm run build.

For deployment, we have step-by-step guides for Vercel and GitHub Pages that walk you through every click.

✨ Features

What themes are available? ▼

Changelog.page includes three built-in themes:

  • Minimal — Clean, typography-focused design. Best for text-heavy updates.
  • Cards — Visual, card-based layout with hover effects. Great for feature announcements.
  • Timeline — Vertical timeline with accent dots. Perfect for showing progression over time.

Switch themes by setting the THEME environment variable: THEME=cards npm run build

Can I customize the themes? ▼

Absolutely! Each theme is built with plain HTML and CSS, making customization straightforward:

  • Edit themes/[theme-name]/css/style.css to change colors, fonts, and spacing
  • Modify themes/[theme-name]/template.html to change the layout structure
  • Add custom CSS in the template's <style> block

All themes use CSS custom properties for easy color customization. Change the primary color by updating the --primary variable.

Does it support RSS feeds? ▼

Yes! Changelog.page automatically generates both RSS 2.0 and JSON Feed v1.1 feeds from your Markdown entries.

The feeds include:

  • Entry titles and content
  • Publication dates
  • Categories/tags
  • Author information
  • Links to individual entry pages

Feed auto-discovery meta tags are included in the HTML head, so feed readers can automatically detect your changelog feed.

Can I use my own domain? ▼

Yes! Since you're self-hosting, you can use any custom domain you own. The setup depends on your hosting platform:

  • Vercel: Add your domain in project settings, update DNS records
  • Netlify: Configure in Site settings → Domain management
  • GitHub Pages: Add a CNAME file with your domain

See our Custom Domain Setup guide for step-by-step instructions.

✍️ Writing & Content

What format should I use for entries? ▼

Changelog entries use Markdown with YAML frontmatter. Here's the basic structure:

---
title: "Your Update Title"
date: 2026-04-08
category: feature
---

Your content here in Markdown...

See our Templates for real-world examples and best practices.

What categories are supported? ▼

The default categories are:

  • feature — New features and capabilities
  • improvement — Enhancements to existing features
  • fix — Bug fixes
  • security — Security updates and patches
  • deprecated — Features being removed
  • breaking — Breaking changes

You can customize or add new categories by modifying the theme CSS and templates.

Can I schedule posts for the future? ▼

Yes! Entries with future dates are automatically excluded from the build. Just set the date field to a future date:

---
date: 2026-04-15
---

When you rebuild after that date passes, the entry will be included automatically.

Can I add images to my changelog? ▼

Yes, you can include images using standard Markdown syntax:

![Alt text](./images/screenshot.png)

Place your images in a folder (like images/ or assets/) and reference them with relative paths. Make sure to commit the images to your repository.

🚀 Deployment

Where can I host my changelog? ▼

You can host your changelog on any static hosting platform:

  • Vercel — Excellent for Next.js/Vercel ecosystem (recommended)
  • Netlify — Great drag-and-drop deployment
  • GitHub Pages — Free hosting directly from your repo
  • Cloudflare Pages — Fast global CDN
  • AWS S3 + CloudFront — Enterprise option

Any platform that supports static HTML/CSS/JS will work perfectly.

Can I automate deployments? ▼

Yes! We include a GitHub Actions workflow that automatically deploys your changelog when you push changes. The workflow:

  1. Triggers on every push to the main branch
  2. Installs dependencies
  3. Builds the changelog
  4. Deploys to your configured platform

See the .github/workflows/deploy.yml file in the generator directory.

How do I preview changes before deploying? ▼

Run the local development server:

npm run serve

This starts a local server at http://localhost:3000 where you can preview your changelog. Changes to Markdown files are automatically reflected (you may need to refresh the browser).

🔧 Troubleshooting

My changes aren't showing up. What's wrong? ▼

Check these common causes:

  1. Did you rebuild? Run npm run build after making changes
  2. Is the date in the future? Future-dated entries are excluded
  3. Is the file in the right folder? Markdown files should be in content/
  4. Is the frontmatter valid? Check for YAML syntax errors
  5. Is there a build error? Check the console output for errors
The build is failing. How do I debug? ▼

First, check the error message in your terminal. Common issues:

  • Invalid YAML frontmatter — Make sure dates are in YYYY-MM-DD format
  • Missing dependencies — Run npm install
  • Syntax errors in Markdown — Check for unclosed tags or invalid characters

Run the build with verbose output for more details:

DEBUG=true npm run build
How do I update to the latest version? ▼

If you cloned the repository:

git pull origin main
npm install

If you downloaded a release, download the latest release and copy your content/ folder to the new version.

Always back up your content/ folder before updating, as theme files may be overwritten.
