Scheduled Posts

Write your changelog entries now, publish them automatically later. Perfect for planning product launches, feature announcements, and maintaining a consistent publishing schedule.

How Scheduling Works

Changelog.page supports future-dated entries. When you set a date in the future, the entry will be automatically hidden from your changelog until that date arrives. Once the date passes and your site rebuilds, the entry appears instantly.

📅 Automatic Publishing

Future-dated entries are filtered out at build time. Set up a daily scheduled build (via GitHub Actions or similar) to automatically publish entries when their date arrives.

Creating a Scheduled Entry

  1. Create your entry file in the content/ directory with a future date in the filename: 
    2026-04-15-new-feature-launch.md
  2. Add your frontmatter with the matching future date: 
    ---
title: "Exciting New Feature Launch"
date: 2026-04-15
category: Feature
tags: [launch, new-feature]
---

Your announcement content here...
  3. Commit and push the file to your repository. The entry will be stored but not visible until the date arrives.
  4. Automatic publication happens when your next build runs on or after the scheduled date.

Date Sources

The build system checks for dates in the following order:

Source Format Example
Frontmatter (recommended) YYYY-MM-DD date: 2026-04-15
Filename YYYY-MM-DD-*.md 2026-04-15-feature.md

💡 Pro Tip

Always include the date in frontmatter for scheduled posts. This ensures the exact publication date is preserved even if the filename changes.

Setting Up Automatic Publishing

To automatically publish scheduled entries, set up a scheduled build workflow:

GitHub Actions (Recommended)

Add this workflow to .github/workflows/scheduled-build.yml:

name: Daily Scheduled Build

on:
  schedule:
    # Run daily at 9:00 AM UTC
    - cron: '0 9 * * *'
  workflow_dispatch:  # Allow manual trigger

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          
      - name: Install dependencies
        run: npm ci
        
      - name: Build changelog
        run: npm run build
        
      - name: Deploy to hosting
        # Add your deployment step here

Build Frequency Recommendations

Schedule Cron Best For
Hourly 0 * * * * High-frequency updates, news sites
Daily (9 AM) 0 9 * * * Most SaaS products (recommended)
Weekly (Monday) 0 9 * * 1 Low-frequency updates, batch releases

Use Cases

🚀 Product Launches

Write your launch announcement in advance, schedule it for launch day, and focus on the launch itself. No last-minute writing required.

📋 Content Calendar

Plan your changelog content weeks ahead. Batch-write entries during quiet periods, schedule them for consistent publishing.

🌍 Multi-Timezone Teams

Schedule announcements to go live at optimal times for your primary audience, regardless of where your team is located.

🔒 Embargoed Updates

Prepare changelog entries for security patches or sensitive updates. They'll only appear once the embargo lifts.

Testing Scheduled Entries

To preview scheduled entries before their publication date:

  1. Run a local build with the current date overridden:
# Preview as if it's April 15, 2026
NODE_ENV=development date -s "2026-04-15" npm run build

# Or modify the date in your local environment
FAKE_DATE=2026-04-15 npm run build

⚠️ Important Note

Scheduled entries rely on your build system having the correct date/time. If you're using a CI/CD service, ensure its clock is accurate. Entries with future dates will not appear until the build runs on or after that date.

Best Practices

  • Use consistent scheduling — Pick a build frequency and stick to it so readers know when to expect updates.
  • Buffer time for review — Schedule entries at least a few hours after you expect to finish writing them.
  • Preview before scheduling — Run a local build to verify formatting and content before committing.
  • Document your schedule — Let your team know when builds happen so they understand when changes go live.
  • Use descriptive filenames — 2026-04-15-api-v2-launch.md is clearer than 2026-04-15-update.md.

Troubleshooting

Entry not appearing on scheduled date

  • Verify the date format is YYYY-MM-DD
  • Check that a build has run on or after the scheduled date
  • Ensure your CI/CD system's timezone matches your expectations
  • Check the build logs for any date-related errors

Entry appeared too early

  • The build system uses UTC dates by default
  • If you specified a local date, it may publish when that date hits UTC
  • Use explicit dates in frontmatter to avoid ambiguity
