Open Source Changelogs: Best Practices for GitHub Projects

Learn how maintainers of popular open source projects craft changelogs that keep contributors engaged and users informed. Complete with templates and real-world examples.

A well-maintained changelog is one of the most valuable yet underappreciated assets in open source software. It's the bridge between maintainers pouring hours into code and users understanding the value of each release. Yet many GitHub projects either skip changelogs entirely or treat them as afterthoughts—a missed opportunity to build trust and community.

In this guide, we'll examine how successful open source projects handle changelogs, from React to VS Code. You'll learn the conventions that make changelogs useful for contributors, the formats that scale with your project, and how to automate the process without losing the human touch.

Why Open Source Changelogs Matter More

Open source projects face unique communication challenges. Your users range from casual developers who just want to know if they should upgrade, to power users who need to understand breaking changes before they migrate, to contributors deciding whether to invest their time in your project.

A good changelog serves all of these audiences:

  • For users: It answers "Should I upgrade?" and "What will break?"
  • For contributors: It shows the project is actively maintained and documents what work is valued
  • For maintainers: It reduces duplicate issues and support questions about changes
  • For decision-makers: It demonstrates project maturity when evaluating dependencies

Real Example: React's Changelog Approach

React maintains separate changelogs for different audiences:

  • CHANGELOG.md — Technical details for contributors
  • Blog posts — User-friendly release announcements with context
  • Release notes — Summaries on GitHub Releases for quick scanning

This multi-layer approach ensures everyone gets the information they need in the format they prefer.

The Keep a Changelog Standard

The most widely adopted changelog format in open source is Keep a Changelog. It provides a simple, human-readable structure that has become the de facto standard for GitHub projects.

Core Principles

  1. Changelogs are for humans, not machines. Write for your users, not your parser.
  2. There should be an entry for every single version. No gaps, no matter how small the change.
  3. Same types of changes should be grouped. Use categories like Added, Changed, Deprecated, Removed, Fixed, and Security.
  4. Versions and sections should be linkable. Use Markdown headers that generate anchors.
  5. The latest version comes first. Reverse chronological order makes scanning easier.
  6. Include release dates. ISO 8601 format (YYYY-MM-DD) works everywhere.

The Standard Format

# Changelog

All notable changes to this project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Added
- New features that have been added

### Changed
- Changes in existing functionality

### Deprecated
- Soon-to-be removed features

### Removed
- Now removed features

### Fixed
- Any bug fixes

### Security
- Security improvements

## [1.0.0] - 2026-04-08

### Added
- Initial release with core functionality
- Support for Markdown rendering
- Three built-in themes

[Unreleased]: https://github.com/user/repo/compare/v1.0.0...HEAD
[1.0.0]: https://github.com/user/repo/releases/tag/v1.0.0

💡 Pro Tip

The "Unreleased" section is your friend. Keep it updated as you merge PRs, then simply rename it when you cut a release. This ensures you never forget what changed.

Semantic Versioning: The Companion Standard

Changelogs work best when paired with Semantic Versioning (SemVer). The structure of your version numbers should tell users what to expect:

Version Format What It Means Changelog Section
MAJOR X.y.z Breaking changes—users must modify their code Removed, Changed (breaking)
MINOR x.Y.z New features—backwards compatible Added, Changed (non-breaking)
PATCH x.y.Z Bug fixes—backwards compatible Fixed, Security

Learning from Popular Projects

VS Code: Detailed but Scannable

VS Code's changelog is a masterclass in organization. Each release is massive, yet easy to navigate because of their consistent structure:

  • Executive summary at the top for users who just want highlights
  • Theme-organized sections (Workbench, Editor, Terminal, etc.)
  • Issue/PR references for every change
  • Contributor acknowledgments to build community

VS Code Release Note Style

"Preview: Drag and drop problems, search results, and symbols. This milestone we're enabling new drag and drop features in the editor...

Contributions to drag and drop: @bpasero, @mjbvz (#12345)"

Node.js: Security-Focused

Node.js handles changelogs with the seriousness you'd expect from infrastructure software:

  • Separate changelogs for each release line (LTS strategy)
  • Security releases get dedicated sections with CVE references
  • Breaking changes are explicitly called out with migration guides
  • Deprecation notices include timeline for removal

Vue.js: Community-Centric

Vue's changelog reads like a community newsletter:

  • Friendly tone that acknowledges contributor efforts
  • Links to documentation updates alongside code changes
  • Migration guides for breaking changes
  • Explicit thanks to sponsors and significant contributors

Writing Great Changelog Entries

The format is just the container. What you write inside matters more. Here's how to craft entries that actually help your users:

Start with the User Impact

Bad: "Refactored the WidgetManager class"

Good: "Widget loading is now 3x faster for pages with 50+ widgets"

Include Migration Instructions

For breaking changes, always include before/after examples:

### Changed
- The `formatDate()` function now requires a locale parameter
  
  **Migration:**
  ```javascript
  // Before
  formatDate(date)
  
  // After
  formatDate(date, 'en-US')
  ```

Reference Issues and PRs

Linking to issues and pull requests provides context for users who want to dig deeper:

### Fixed
- Memory leak in long-running processes (#456, thanks @contributor)

Use Consistent Categories

Stick to the standard categories but add project-specific ones as needed:

  • Added — New features
  • Changed — Changes to existing functionality
  • Deprecated — Soon-to-be removed features
  • Removed — Now removed features
  • Fixed — Bug fixes
  • Security — Security fixes
  • Performance — Speed improvements (optional)
  • Documentation — Docs changes only (optional)

Automation Without Losing Humanity

As your project grows, maintaining changelogs manually becomes burdensome. But automation shouldn't mean robotic release notes. Here's how to strike the balance:

GitHub Releases API

The simplest automation is using GitHub's built-in release notes generator:

  1. Go to Releases → Draft a new release
  2. Click "Generate release notes"
  3. Edit the auto-generated content to add context
  4. Publish

This pulls in PR titles and contributor information, giving you a starting point that you can humanize.

Conventional Commits

The Conventional Commits specification allows tools to automatically categorize changes:

feat: add user authentication
fix: resolve memory leak in data processing
docs: update API reference
BREAKING CHANGE: drop support for Node 14

Tools like semantic-release can then automatically generate changelogs and version bumps based on these commit messages.

⚠️ Warning

Conventional Commits work best when enforced. Consider using commitlint or pre-commit hooks. Without discipline, you'll end up with uncategorized changes that require manual cleanup.

Handling Special Cases

Security Releases

Security fixes deserve special treatment:

  • Create a dedicated Security section in your changelog
  • Include CVE identifiers when applicable
  • Document affected versions clearly
  • Provide workarounds for users who can't upgrade immediately
  • Credit security researchers who reported the issue

Breaking Changes

Breaking changes are the most important entries in your changelog:

  • Lead with the impact: "Applications using X will need to update Y"
  • Provide clear migration steps
  • Link to detailed migration guides for complex changes
  • Indicate the timeline for deprecated features

Deprecation Notices

Give users plenty of warning before removing features:

### Deprecated
- The `legacyAuth()` method is deprecated and will be removed in v3.0.0
  - Use `newAuth()` instead
  - See migration guide: https://docs.example.com/migration

Common Mistakes to Avoid

1. Commit Logs as Changelogs

Commit messages are for developers during development. Changelogs are for users after release. They're different audiences with different needs.

2. Vague Entries

"Fixed bugs" or "Improved performance" tells users nothing. Be specific about what was fixed and how performance improved.

3. Missing Migration Guides

Every breaking change should include clear migration instructions. Assume your users are busy and frustrated—they need help, not a puzzle.

4. Inconsistent Formatting

Pick a format and stick to it. Inconsistency makes scanning difficult and suggests the project lacks attention to detail.

5. Forgetting the Unreleased Section

The gap between releases is where changelogs fall apart. Keep an Unreleased section updated as you merge changes.

Measuring Changelog Success

How do you know if your changelog is working? Track these indicators:

  • Upgrade velocity: Are users upgrading promptly after releases?
  • Support questions: Are you getting fewer "what changed?" questions?
  • Issue quality: Are bug reports more specific after users read changelogs?
  • Contributor engagement: Do contributors reference changelogs when proposing changes?

Key Takeaways

Maintaining a great changelog for your open source project isn't just documentation—it's community building. Here's what to remember:

  1. Adopt Keep a Changelog format — It's the standard for a reason
  2. Use Semantic Versioning — Communicate compatibility clearly
  3. Write for users, not machines — Focus on impact, not implementation
  4. Automate where it helps — But don't lose the human touch
  5. Treat breaking changes with care — Always provide migration paths
  6. Maintain an Unreleased section — Don't let changes pile up
  7. Credit contributors — Changelogs are community celebration too

The best open source projects treat changelogs as first-class artifacts. They're not afterthoughts—they're an essential part of the release process that respects your users' time and attention.
